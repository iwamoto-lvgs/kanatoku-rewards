## 0. このアプリを一言で

かなトク（神奈川県のキャッシュレス還元キャンペーン）の還元上限は **決済サービスごと** に設定される。複数の決済アプリを併用すると「どのサービスであと何円分の還元を受けられるか」をアプリ横断で把握できない。本アプリは支払いを **手動入力** し、サービスごとの残枠を可視化して取りこぼし・無駄打ちを防ぐ、サーバー不要の単一HTML WEBアプリ。

## 1. キャンペーン要項（計算の前提・出典）

出典: かながわトクトクキャンペーン！『かなトク！』公式サイト
https://www.kanatokucpn.pref.kanagawa.jp/

| 項目 | 値 |
|---|---|
| 対象決済 | AEON Pay / au PAY / d払い / PayPay / メルペイ / 楽天ペイ の6種 |
| 還元率 | 中小・小規模店 **20%** / 大手店 **10%**（店舗により異なる） |
| サービスあたり付与上限 | **2,500円** 相当 |
| 1回の支払いあたり付与上限 | **1,500円** 相当 |
| 超過分 | 付与されない（切り捨て） |
| 期間 | 2026年6月19日〜予算上限到達まで（早期終了あり） |

これらの数値は要項由来のハードコード。要項変更時は定数を更新する想定。

## 2. 計算ロジック（実装の核・最重要）

1件の支払いに対する **実付与額** を必ず次の順序で算出する。**この順序を変更しないこと。**

```
理論付与額       = 支払額 × 還元率（0.20 または 0.10）
1回上限適用      = min(理論付与額, 1500)
サービス上限適用 = min(1回上限適用, 2500 − そのサービスの既付与累計)
実付与額         = max(0, サービス上限適用)
```

ポイントは **上限超過を自動で頭打ちにすること**。「1回1,500円上限」と「サービス累計2,500円上限」の二段の丸めがかかる。

### 計算ロジックの実装要件

- UI・保存層から独立した **純粋関数** として実装する（テスト・将来のBaaS移行のため）。
- 還元の定数（2500 / 1500 / 0.20 / 0.10）はマジックナンバーで散らさず、コード冒頭に定数として集約する。

### 受け入れ用テストケース（必ず通すこと）

| 支払額 | 還元率 | 既付与累計 | 期待実付与額 | 説明 |
|---|---|---|---|---|
| 10000 | 0.20 | 0 | 1500 | 1回上限で丸め（理論2000→1500） |
| 5000 | 0.20 | 0 | 1000 | 上限未満そのまま |
| 10000 | 0.10 | 0 | 1000 | 10%店 |
| 5000 | 0.20 | 2000 | 500 | サービス残枠で丸め（残500） |
| 5000 | 0.20 | 2500 | 0 | 上限到達済み |
| 7500 | 0.20 | 0 | 1500 | 20%店で1回上限ちょうど |
| 15000 | 0.10 | 0 | 1500 | 10%店で1回上限ちょうど |

### 派生表示値

- サービスごとの **残り還元枠** = `2500 − 実付与累計`
- 残枠を使い切るのに必要な追加支払額（参考値）
  - 20%店: `残枠 ÷ 0.20`
  - 10%店: `残枠 ÷ 0.10`
  - ※ダッシュボードでは20%店/10%店の両方を併記する

## 3. 機能要件

### 3.1 ユーザー（プロフィール）管理
- **複数人を分けて管理**（家族など）。ユーザーの追加・名前編集・削除ができる。
- 画面上部でアクティブユーザーを切り替え、すべての集計・記録・入力がアクティブユーザーに紐づく。

### 3.2 支払い記録の登録
入力項目:
- ユーザー（アクティブユーザーが既定）
- 決済サービス（6種から選択）
- 支払額（円）
- 還元率（20% / 10% から選択）
- 日付（既定は当日）
- メモ（任意。店名など）

登録時に第2章のロジックで実付与額を自動計算して保存・表示する。
**「1回上限で丸められた」「サービス上限に達した」場合はその旨をユーザーに明示する**（例: 「1回上限により1,500円に丸められました」「○○は上限に達しました」）。

### 3.3 記録の編集・削除
- 登録済みレコードの修正・削除。変更時は累計と残枠を再計算する。

### 3.4 ダッシュボード（メイン画面）
アクティブユーザーについて、6サービス分のカードを表示:
- サービス名
- 付与済み累計 / 2,500円（**プログレスバー**）
- 残り還元枠
- 残枠を使い切る目安支払額（20%店 / 10%店の両方を併記）
- 上限到達サービスには **達成バッジ** を表示

全体サマリー:
- そのユーザーの **総付与額**（最大15,000円 = 2,500 × 6サービス）
- 全サービス上限到達までの進捗

### 3.5 履歴一覧
- 記録を **日付降順** で一覧表示。
- **サービス・ユーザーでフィルタ** できる。

## 4. 非機能要件

- **構成**: サーバー不要。ブラウザ単体で完結する **単一ファイル（index.html に HTML/CSS/JS をまとめる）**。
- **技術スタック**: バニラ HTML / CSS / JavaScript。フレームワーク・ビルドツール不使用。外部依存・CDN依存は極力避ける。
- **対象環境**: スマホ・PCのモダンブラウザ。**モバイルファースト**のレスポンシブ。
- **オフライン動作**: ネット接続不要。

### データ保存方針（確定事項）
- **保存先**: 端末のブラウザの **localStorage**（永続化）。サーバーは持たない。
- **保存スコープ**: 端末・ブラウザごとに独立。複数端末間では自動共有されない。
- **保存キー命名**: プレフィックス `kanatoku:` を付ける（例 `kanatoku:data`）。
- **アクセス抽象化**: localStorage を直接触らず、`loadData()` / `saveData()` 等のラッパー関数経由に統一する（将来のBaaS移行を容易にするため。第7章参照）。
- **バックアップ/移行**: **JSONエクスポート/インポート** 機能を備える（機種変更・別端末移行・手動共有を兼ねる）。
- **注意喚起**: localStorage はブラウザのデータ削除やプライベートモードで消える可能性があるため、定期的なJSONエクスポートを推奨する旨をUI/READMEに記載する。

## 5. データモデル（確定）

```
User {
  id,            // 一意ID
  name           // 表示名
}

Payment {
  id,            // 一意ID
  userId,        // User.id への参照
  service,       // aeonpay | aupay | dpay | paypay | merpay | rakutenpay
  amount,        // 支払額（数値）
  rate,          // 0.20 | 0.10
  date,          // 日付
  memo,          // 任意（店名など）
  grantedAmount  // 計算結果（保存 or 都度算出のどちらでもよいが、累計計算は実付与額ベースで行う）
}
```

サービスの識別子（固定）:
`aeonpay`, `aupay`, `dpay`, `paypay`, `merpay`, `rakutenpay`
表示名はそれぞれ AEON Pay / au PAY / d払い / PayPay / メルペイ / 楽天ペイ。

## 6. スコープ外（実装しないこと）

- キャンペーン予算上限（県の180億円）到達による **早期終了の検知**。アプリでは不可。期間は手動確認前提。
- 対象外店舗・商品（コンビニ、たばこ、金券、納税等）の **判定**。入力は対象取引である前提とする。
- 付与時期・有効期限の **スケジュール管理**。本アプリは上限管理に特化する。

---

## 7. 将来の拡張方針（実装は不要、設計で配慮するのみ）

現時点は localStorage 単独で運用。複数人・複数端末での共有が必要になったときに備え、**データモデルは変えず保存層のみ差し替えられる設計**にしておく。

移行パス:
1. 第1段階（現行）: localStorage + JSONエクスポート/インポート。
2. 第2段階: 共有要求が出たら BaaS（Supabase / Firebase Firestore）へ。GitHub Pages のままフロントだけで完結。認証・アクセス制御の設計が必要。
3. 第3段階: さらに自由度が必要なら自前バックエンド（Cloudflare Workers / Vercel Functions）+ DB。本規模では過剰で原則第2段階で完結。

移行容易性のための設計上の留意点（今の実装で守ること）:
- データ読み書きはラッパー関数（`loadData()` / `saveData()`）経由にする。
- データ構造は id 正規化・`userId` 参照を維持する。
- 計算ロジックは保存層に依存させない（純粋関数として独立）。
